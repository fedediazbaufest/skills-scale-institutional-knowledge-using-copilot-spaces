# OctoAcme — Retrospective & Continuous Improvement

## Purpose
Capture learnings and convert them into actionable improvements.

## When
After each sprint, release, or important milestone. Also after incidents.

## Structure
- What went well
- What could be improved
- Action items (owner, due date)
- Follow-up on previous action items
- Feedback from Support/Customer Success on user-reported issues since last release

## Running a Retrospective
- Timebox: 45–75 minutes depending on team size
- Use an anonymous idea board if needed to encourage candor
- Prioritize 2–3 top action items to avoid overload

## Tracking Improvements
- Add action items to the project backlog or issues with clear owners and timelines
- Review outstanding actions in the weekly PM sync

## Continuous Improvement Culture
- Measure impact of action items
- Celebrate improvements and make small, iterative changes
- Security Champion tracks and closes security-related action items from retrospectives
- QA Lead monitors quality trends (defect rates, test coverage) and proposes improvements
- Support/Customer Success shares patterns from user feedback to inform backlog prioritization

## Retrospective Participation Guide
All roles should participate in or contribute to retrospectives:
| Role | Contribution |
| --- | --- |
| Project Manager | Facilitates session, captures and tracks action items |
| Product Manager | Shares outcome metrics and customer impact data |
| Technical Lead | Highlights architectural improvements or technical debt actions |
| Developers | Discusses what slowed delivery and what could improve |
| UX/UI Designer | Shares usability feedback and design process improvements |
| QA Lead | Presents defect trends and test coverage findings |
| Security Champion | Reports on security findings and mitigation follow-ups |
| Support/Customer Success | Brings user-reported issues and post-release feedback |
